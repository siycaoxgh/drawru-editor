# v3.6 —— 上一阶段稳定化版本

## 版本定位

v3.6 是 v3.5.x 稳定化工作的收束版本，重点是微信兼容、安全边界、图片比例和 Footer 渲染链路统一，不新增在线发布或复杂图片管理功能。

## 主要完成内容

### 1. 统一渲染链路

```text
Markdown → Parser → AST/Nodes → enrich-image → Renderer → Clipboard
```

- 标题、段落、图片、引用、信息、Divider、Footer 组件化。
- 保留 `default`、`software` Theme。
- Footer 固定顺序：`END → Copyright → Recommend → Social → Interaction`。
- 旧 `footer/index.js` 保留为兼容入口，内部转入统一 Footer Pipeline。

### 2. 正文图片与 Header 图片

正文图片和 Header 图片均保持自然比例，不使用 Banner 裁剪逻辑：

```css
width: 100%;
max-width: 100%;
height: auto;
display: block;
```

正文图片继续保留 `section` 容器、圆角、虚线边框、`raw-image`、`crossorigin="anonymous"`，并输出 `data-ratio`、`data-w`、`_width`、`data-s` 等微信兼容元数据。

正文图片禁止使用固定 16:9、`padding-top`、绝对定位和 `object-fit: cover`。Header 图片使用独立 `HeaderImagesRenderer`，保持原始比例。

### 3. 推荐阅读图片

推荐阅读是独立视觉卡片，使用 16:6 比例，允许 `object-fit: cover`：

```css
aspect-ratio: 16 / 6;
width: 100%;
height: 100%;
object-fit: cover;
```

已移除 `padding-top: 37.5%` 的旧占位方案，解决点击微信保存后产生空白区域的问题。

配置面板中，推荐标题和图片 COS 链接必须同时填写；只填写图片链接而没有标题时，该推荐项不会进入预览。编辑器会在应用配置时提示并跳过未填写完整的推荐项。

### 4. Divider 微信兼容

所有文章结构分割线统一调用 `DividerComponent.render()`，不再由各模块自行拼接分割线。组件使用 `<section>` 和零尺寸 SVG 占位符，避免微信重新解析 inline 元素后出现点线错位：

```html
<svg viewBox="0 0 1 1" style="float:left;line-height:0;width:0;vertical-align:top;"></svg>
```

正文内容分割线和文章尾部分割线均遵循同一兼容方案。

### 5. Footer Pipeline

文章尾部固定为：

```text
正文
↓
END
↓
版权说明
↓
推荐文章
↓
平台关注
↓
点赞 / 收藏 / 在看 / 留言
```

版权类型支持：原创、转载、AI生成、整理，不再硬编码为“原创”。社交数据复用 `assets/footer/socials.js`，平台关注区域增加统一介绍文本，互动区域删除“动动小手”。

### 6. 安全边界

- Markdown 段落、标题、引用、信息字段统一转义。
- 图片 URL 仅允许 HTTP/HTTPS 和 PNG/JPEG/GIF/WebP 数据图片。
- 推荐、Header、Social、旧 Footer 入口统一过滤图片协议。
- 图片 `width`、`height` 在增强层统一校验为正数或 `null`。
- 链接 URL 过滤危险协议，并补充 `rel="noopener noreferrer"`。

## 发布前验证

- 非归档 JavaScript 文件语法检查：37 个文件，0 个错误。
- 正文图片自然比例输出检查通过。
- 推荐卡片无 `padding-top:37.5%` 空白占位。
- Divider `<section>` 闭合和零尺寸 SVG 占位检查通过。
- XSS、图片协议和图片尺寸专项检查通过。
- 推荐文章有完整标题和图片、缺少标题、缺少图片三种配置逻辑已覆盖。

## 已知非阻断事项

1. `components/interaction.js` 和 `assets/article/recommend-renderer.js` 是旧兼容实现，当前入口仍加载但主渲染链路不调用。
2. 配置面板图片预览接受范围较宽，最终渲染器会再次执行更严格的图片 URL 白名单。
3. Asset Registry 作为浏览器兼容层保留；图片上传和 COS 管理不属于 v3.6 范围。

# v3.5 —— 生产化稳定阶段

v3.5.x 的核心目标是让 DrawRu 从“浏览器里显示正常”进一步达到“复制到微信公众号后稳定”。

## v3.5.0 —— 编辑器架构重构

- 建立 Markdown → AST/Nodes → Renderer 架构。
- 拆分标题、段落、图片、引用、Divider、Footer 等组件。
- 建立 `default`、`software` Theme 系统。

## v3.5.1 —— 微信兼容优化

- 参考秀米 HTML，强化 `<section>`、inline style 和 `raw-image` 结构。
- 优化图片的 `crossorigin="anonymous"` 和微信复制兼容属性。
- 逐步减少复杂 `div`、外部样式和浏览器专用布局依赖。

## v3.5.2 —— Footer 系统建设

- 拆分 END、版权、推荐、社交和互动模块。
- 为后续固定 Footer Pipeline 和稳定排序打基础。

## v3.5.3 —— 图片节点增强

- 图片节点增加尺寸、对齐、边框和标题等扩展属性。
- 引入 `enrich-image` 数据增强层。

## v3.5.4 —— 图片资产管理

- 建立 Asset Registry、store、validator、resolver 和 PicGo 导入相关能力。
- 支持 `asset:id` 内部引用。
- 明确资产系统不负责上传和 COS 管理。

## v3.5.5-beta —— 微信生产化稳定

- 完成 COS 图片复制链路和微信重新请求兼容处理。
- 参考秀米结构优化图片和 Footer HTML。
- 修复固定比例容器造成的图片裁剪、空白和微信保存后比例异常。
- 统一 Divider，使用 SVG 空占位稳定微信基线。
- 稳定 Renderer → currentHTML → Clipboard → 微信公众号复制链路。

# 历史版本：v3.4 → v2.0

以下版本按从新到旧排列，用于说明 DrawRu 的架构演进背景。

## v3.4 —— 架构稳定前阶段

- 初步分离 Parser、Node 和 Renderer。
- 引入 Theme 概念，开始分离文章结构与视觉样式。
- 持续优化 `<section>`、图片格式和 inline style 的微信兼容性。

遗留问题：Parser 与 Renderer 仍有耦合，图片能力、Footer 数据化和模板扩展能力不足，这些问题推动了 v3.5 的架构重构。

## v3.3 —— Footer 重构阶段

Footer 从混合的单体逻辑开始拆分为 END、版权、社交、关注和相关推荐等模块，为后续 Pipeline 奠定基础。

## v3.2 —— 组件化探索阶段

建立 `components/` 目录，将标题、段落、图片、Divider 和 Footer 等逻辑拆成独立组件，提高维护和扩展能力。

## v3.1 —— 秀米风格探索阶段

开始系统研究 `reference/xiumi-original.html`，逐步采用 `<section>`、inline style、`raw-image`，并减少复杂 CSS 对微信公众号复制结果的影响。

## v3.0 —— 文章结构优化阶段

公众号排版从“生成一段 HTML”转向“生成完整文章结构”：标题、小标题、图片和分隔线开始以章节组件组织。

## v2.0 —— 单文件编辑器阶段

DrawRu 最初是 Markdown 到微信公众号 HTML 的快速转换工具：

```text
Markdown → 简单解析 → HTML 字符串拼接 → 复制到公众号
```

当时已支持标题、小标题、正文、图片、列表、分隔线、基础模板和富文本复制，但样式、内容、模板和 Footer 大量集中在 `app.js`，为后续架构拆分留下了改进空间。

## 一句话总结

> v2.0 是“能用”，v3.0–v3.4 是“开始像一个项目”，v3.5 是“成为一个完整工具”，v3.6 则是面向发布的稳定收束版本。
